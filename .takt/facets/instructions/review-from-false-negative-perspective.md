# 偽陰性パターン検出レビュー詳細手順

## 事前確認

1. **コードの動作理解**
   - `git diff` で変更内容を確認
   - 実装されたロジックの動作フローを理解
   - 入力と出力の関係を把握
   - エッジケースや特殊な条件を想定

2. **実行環境の確認**
   - 本番環境と開発環境の違いを確認
   - 並行処理や非同期処理の有無を確認
   - 外部依存（DB、API、ファイルシステム等）を確認

## レビュー実施チェックリスト

### 1. エッジケース処理の漏れチェック

- [ ] **境界値の扱い**
  - 配列・リストの最初/最後の要素処理は正しいか
  - 数値の最小値/最大値の処理は適切か
  - インデックスの範囲外アクセスの可能性はないか
  - オーバーフロー/アンダーフローの考慮があるか
- [ ] **空データの扱い**
  - 空配列、空文字列、空オブジェクトの場合の動作は適切か
  - null、undefined、None の場合のハンドリングは十分か
  - 0除算の可能性がないか
  - 空のIterableやOptionalの扱いは適切か
- [ ] **特殊な入力値**
  - 負の数、小数点以下の値の扱いは正しいか
  - 特殊文字（改行、タブ、Unicode文字、絵文字等）の扱いは適切か
  - 極端に長い文字列や大きなデータの処理は考慮されているか
  - 不正な形式のデータに対する処理は適切か

### 2. エラーハンドリングの不備チェック

- [ ] **例外処理の網羅性**
  - すべての例外型が適切にキャッチされているか
  - 例外を握りつぶしていないか（空のcatchブロック等）
  - エラー時のリソース解放（finally、defer、using等）が適切か
  - ネストされた例外処理が正しく動作するか
- [ ] **エラーの伝播**
  - エラーが適切に上位レイヤーに伝播されているか
  - エラー情報（スタックトレース、コンテキスト）が失われていないか
  - ユーザーに分かりやすいエラーメッセージが返されるか
  - エラーログが適切に出力されているか
- [ ] **リトライ・フォールバック**
  - 一時的な障害に対するリトライ処理が必要ではないか
  - リトライ回数や間隔が適切か
  - フォールバック処理（代替処理）が適切か
  - サーキットブレーカーパターンの必要性はないか

### 3. 競合状態・タイミング問題チェック

- [ ] **並行処理の安全性**
  - 共有リソースへのアクセスが排他制御されているか
  - データ競合（Data Race）の可能性はないか
  - ロックの取得・解放が適切か
  - デッドロックやライブロックの可能性はないか
- [ ] **非同期処理**
  - Promise/Future の適切な待機処理があるか
  - async/await の使い方は正しいか
  - コールバック地獄になっていないか
  - 非同期処理のエラーハンドリングが適切か
- [ ] **タイミング依存**
  - 処理順序の前提が崩れた場合の考慮があるか
  - 初期化前のアクセスの可能性はないか
  - タイムアウト処理が適切か

### 4. リソースリークチェック

- [ ] **メモリリーク**
  - 不要になったオブジェクトへの参照が残っていないか
  - イベントリスナーの登録解除が適切か
  - クロージャによるメモリ保持の問題はないか
  - 循環参照が発生していないか
- [ ] **ファイルハンドル・接続リーク**
  - ファイルが適切にクローズされているか
  - データベース接続が解放されているか
  - ネットワーク接続が適切にクローズされているか
  - リソースの取得と解放が対になっているか（RAII、using、defer等）

### 5. セキュリティ脆弱性チェック

- [ ] **インジェクション攻撃**
  - SQLインジェクションの可能性はないか
    - パラメータ化クエリ（Prepared Statement）を使用しているか
    - ORMの使い方は安全か
  - XSS（クロスサイトスクリプティング）の可能性はないか
    - ユーザー入力のエスケープ処理が適切か
    - innerHTML等の危険なAPIの使用は適切か
  - コマンドインジェクションの可能性はないか
    - シェルコマンドの引数が適切にエスケープされているか
  - LDAP、XML、XPathインジェクションの可能性はないか
- [ ] **認証・認可**
  - 認証チェックが適切に行われているか
  - 権限チェックが漏れていないか
  - セッション管理は安全か
  - CSRF対策が適切か
- [ ] **機密情報の扱い**
  - パスワード、APIキー等がハードコードされていないか
  - 機密情報がログに出力されていないか
  - 暗号化が必要なデータが平文で扱われていないか
  - 環境変数やシークレット管理が適切か
- [ ] **入力検証**
  - ユーザー入力の検証（バリデーション）が十分か
  - ホワイトリスト方式の検証を使用しているか
  - サニタイゼーション処理が適切か
  - ファイルアップロードの検証（サイズ、拡張子、MIME type）が十分か

### 6. NULL/undefined チェック漏れ

- [ ] **デリファレンス前のチェック**
  - null/undefined の可能性がある変数を参照する前にチェックしているか
  - Optional Chaining (`?.`) や Null Coalescing (`??`) が適切に使用されているか
  - Null Object パターンの使用は適切か
- [ ] **関数の戻り値**
  - null/undefined を返す可能性がある関数の戻り値チェックがあるか
  - Optional型の適切な扱い（unwrap前のチェック）
  - Result型やEither型の活用は適切か

### 7. 型安全性の問題チェック

- [ ] **型キャスト・型アサーション**
  - 型キャストが安全か（実行時エラーの可能性）
  - any型やunknown型の使用が最小限か
  - 型ガード（Type Guard）が適切に使われているか
  - 型アサーション（as）の使用が正当化できるか
- [ ] **暗黙の型変換**
  - 予期しない型変換が発生していないか
  - 比較演算子（==と===）の使い分けは適切か
  - 数値と文字列の演算が適切か

### 8. ロジックの正確性チェック

- [ ] **条件分岐の漏れ**
  - すべてのケースがカバーされているか
  - else節や default節が適切に実装されているか
  - 論理演算子（AND/OR）の優先順位は正しいか
  - 短絡評価（Short-circuit evaluation）の理解は正しいか
- [ ] **ループの正確性**
  - ループの終了条件は正しいか
  - オフバイワンエラー（Off-by-one Error）はないか
  - 無限ループの可能性はないか
  - ループ変数の変更が適切か
- [ ] **計算の正確性**
  - 浮動小数点演算の誤差は考慮されているか
  - 演算の順序は正しいか（優先順位、結合規則）
  - 整数オーバーフローの可能性はないか
  - 丸め誤差の影響は考慮されているか

## 検出の重点ポイント

「通常のレビューでは見逃しやすい」以下の観点を重点的にチェック：

### 1. 実装は本当に正しく動作するか？
- 単体テストが通っても実環境で問題が出る可能性
- エッジケースでの動作確認
- 実データでの動作想定

### 2. すべてのエラーケースを処理しているか？
- 正常系だけでなく異常系も確認
- ネットワークエラー、タイムアウト、リソース不足等
- サードパーティAPIのエラーレスポンス

### 3. 型安全性は保証されているか？
- 型システムの穴がないか
- 実行時の型エラーの可能性
- null/undefined の適切な扱い

### 4. セキュリティリスクはないか？
- 攻撃ベクトルの検討
- OWASP Top 10 の観点からチェック
- 機密情報の漏洩リスク

## レビュー結果の出力

### 指摘事項の分類

各指摘事項を以下の重要度で分類：

- **Critical**: セキュリティ脆弱性、データ損失の可能性、システムクラッシュ
- **High**: エッジケース処理の欠如、リソースリーク、重大なロジックエラー
- **Medium**: エラーハンドリングの不足、null チェック漏れ
- **Low**: 軽微な型安全性の問題、潜在的なパフォーマンス問題

### 出力形式

`.tmp/review-results/false-negative.md` に以下の形式で出力：

```markdown
# 偽陰性パターン検出レビュー結果

## 判定
- [APPROVED / NEEDS_FIX]

## サマリー
- 指摘事項数: X件
- 重要度内訳: Critical: X, High: X, Medium: X, Low: X
- リスクレベル: [高/中/低]

## 指摘事項

### [ファイルパス:行番号] タイトル
**重要度**: [Critical/High/Medium/Low]
**カテゴリ**: [エッジケース/エラーハンドリング/並行処理/リソースリーク/セキュリティ/Nullチェック/型安全性/ロジック]
**リスク**: 発生しうる問題の説明

詳細説明（なぜこの問題が見落とされやすいか）

**推奨対応**:
具体的な修正方法（コード例を含む）
```

## 最終確認

- [ ] 見落としやすい問題を的確に指摘できているか
- [ ] 指摘が具体的で再現可能か
- [ ] 修正方法が明確で実行可能か
- [ ] セキュリティリスクの評価が適切か
- [ ] ファイルが正しいパスに出力されているか
