# 過剰実装レビュー詳細手順

## 事前確認

1. **タスク・要件の確認**
   - 関連するIssue、タスク、要件定義を確認
   - 元々の目的とスコープを理解
   - 「何を実現すべきか」を明確にする

2. **実装の全体像把握**
   - `git status` で追加・変更されたファイル一覧を確認
   - `git diff --stat` でコードの行数変更（追加行数/削除行数）を確認
   - 新しい依存関係やライブラリの追加を確認（package.json、requirements.txt等）

## レビュー実施チェックリスト

### 1. YAGNI原則チェック（You Aren't Gonna Need It）

- [ ] **現在使われているか**
  - 追加されたクラス・関数が実際に使用されているか
  - デッドコード（未使用コード）がないか
  - コメントアウトされたコードが残っていないか
- [ ] **将来のための実装がないか**
  - 「将来必要になるかも」という理由だけで追加された機能がないか
  - 現在の要件に含まれていない拡張性がないか
  - プラグイン機構など、現時点で不要な拡張ポイントがないか
- [ ] **過度な抽象化**
  - 1箇所でしか使われていないのに抽象化されているコードがないか
  - インターフェースや抽象クラスが本当に必要か
  - ジェネリクスが過度に使用されていないか

### 2. シンプル性チェック

- [ ] **より簡潔な実装の可能性**
  - 同じ結果をより少ないコードで実現できないか
  - 複雑な条件分岐をシンプルにできないか
  - ネストの深さを減らせないか
- [ ] **デザインパターンの必要性**
  - Factory、Builder、Strategyなどのパターンが本当に必要か
  - 問題の規模に対してパターンが過剰でないか
  - 単純な関数やクラスで十分ではないか
- [ ] **レイヤー分割の適切性**
  - レイヤーが必要以上に多層化していないか
  - 単純な処理に対して中間層が多すぎないか
  - データの変換だけを行うレイヤーがないか

### 3. 依存関係チェック

- [ ] **新規依存の必要性**
  - 追加されたライブラリやパッケージが本当に必要か
  - 既存のライブラリや標準機能で代替可能ではないか
  - 小さな機能のために大きなライブラリを追加していないか
- [ ] **依存関係の複雑さ**
  - モジュール間の依存関係が複雑すぎないか
  - 循環依存が発生していないか
  - 依存関係グラフが理解しやすいか

### 4. スコープ適合性チェック

- [ ] **タスクとの整合性**
  - 元のタスクやIssueで求められている機能以外の実装がないか
  - 「ついでに」追加された機能がないか
  - PRの目的が明確に保たれているか
- [ ] **機能の範囲**
  - 必要最小限の機能に絞られているか
  - 仮説検証に必要な最小構成（MVP）になっているか
  - 「あったら便利」レベルの機能が含まれていないか

### 5. 設定・柔軟性チェック

- [ ] **設定の必要性**
  - 設定ファイルやパラメータが必要最小限か
  - デフォルト値で十分なのに設定可能にしていないか
  - 環境変数や設定項目が多すぎないか
- [ ] **カスタマイズポイント**
  - フックやコールバックが過度に提供されていないか
  - プラグイン機構が本当に必要か
  - 現在の要件で不要なカスタマイズポイントがないか

### 6. 早すぎる最適化チェック

- [ ] **パフォーマンス最適化**
  - 計測なしにパフォーマンス最適化を行っていないか
  - 現時点でパフォーマンス問題が存在しないのに最適化していないか
  - 可読性を犠牲にした最適化がないか
- [ ] **キャッシング・メモ化**
  - キャッシュ機構が本当に必要か
  - キャッシュ管理の複雑さに見合う効果があるか
  - キャッシュの無効化戦略が適切か

### 7. ドキュメント・コメントチェック

- [ ] **コメントの適切性**
  - 自明なコードに対する冗長なコメントがないか
  - コードを読めば分かることをコメントで説明していないか
  - コメントではなくコードの自己文書化で対応できないか
- [ ] **TODOコメント**
  - 将来の拡張のためのTODOコメントが多すぎないか
  - TODOが本当に必要な改善か

## 判断基準（4つの質問）

各実装要素について以下の質問を自問：

1. **今必要か？**
   - 現在のタスク・要件で本当に必要か
   - 将来のためだけの実装ではないか

2. **シンプルにできるか？**
   - より簡潔な方法はないか
   - 複雑さに見合う価値があるか

3. **削除できるか？**
   - なくても動作するコードではないか
   - 本当にこの抽象化は必要か

4. **読みやすいか？**
   - 他の開発者が理解しやすいか
   - メンテナンスコストは適切か

## レビュー結果の出力

### 指摘事項の分類

各指摘事項を以下の重要度で分類：

- **Critical**: 明らかに不要な大規模実装、スコープ外の機能追加
- **High**: YAGNI違反、過度な抽象化、不要な依存追加
- **Medium**: シンプルにできる箇所、過度な設定
- **Low**: 冗長なコメント、軽微な最適化

### 出力形式

`.tmp/review-results/over-implementation.md` に以下の形式で出力：

```markdown
# 過剰実装レビュー結果

## 判定
- [APPROVED / NEEDS_FIX]

## サマリー
- 指摘事項数: X件
- 重要度内訳: Critical: X, High: X, Medium: X, Low: X
- スコープ適合度: [高/中/低]

## 指摘事項

### [ファイルパス:行番号] タイトル
**重要度**: [Critical/High/Medium/Low]
**カテゴリ**: [YAGNI/シンプル性/依存関係/スコープ/設定/最適化/ドキュメント]

詳細説明（なぜ過剰実装と判断したか）

**推奨対応**:
シンプルにする方法の具体的な提案
```

## 最終確認

- [ ] すべての指摘が建設的で理由が明確か
- [ ] よりシンプルな代替案を提示できているか
- [ ] 過剰実装の判断が適切か（必要な柔軟性を削っていないか）
- [ ] ファイルが正しいパスに出力されているか
