# レビュー結果集約手順

## 事前確認

1. **レビュー結果ファイルの存在確認**
   - 4つのレビュー結果ファイルがすべて存在するか確認
   - ファイルが空でないか確認
   - ファイルのフォーマットが正しいか確認

2. **現在のブランチ情報の取得**
   - `git branch --show-current` で現在のブランチ名を取得
   - `git rev-parse --short HEAD` でコミットハッシュを取得（オプション）

## 実施手順

### Step 1: レビュー結果の読み込み

以下の4つのファイルを読み込む：

```
.tmp/review-results/code-quality.md
.tmp/review-results/test-quality.md
.tmp/review-results/over-implementation.md
.tmp/review-results/false-negative.md
```

### Step 2: 各レビュー結果の解析

各ファイルから以下の情報を抽出：

1. **判定** (APPROVED / NEEDS_FIX)
2. **指摘事項数**
3. **重要度内訳** (Critical, High, Medium, Low の件数)
4. **個別の指摘事項リスト**

### Step 3: 総合判定の決定

以下のロジックで総合判定を決定：

```
判定ロジック:
  if any_review == "NEEDS_FIX":
    総合判定 = "NEEDS_FIX"
  else:
    総合判定 = "APPROVED"

重要度による調整:
  if Critical指摘 > 0:
    総合判定 = "NEEDS_FIX"
    リスクレベル = "高"
  else if High指摘 > 0:
    総合判定 = "NEEDS_FIX"
    リスクレベル = "中"
  else if Medium指摘 >= 5:
    総合判定 = "NEEDS_FIX"
    リスクレベル = "低"
  else:
    総合判定 = "APPROVED"
    リスクレベル = "なし"
```

### Step 4: 指摘事項の統合

重複する指摘事項を統合：

1. **重複判定**
   - 同じファイル・同じ行番号の指摘
   - 内容が類似している指摘（自然言語処理または手動判断）

2. **統合方法**
   - より厳しい重要度を採用
   - すべてのレビューの観点を統合
   - タイトルは最も具体的なものを採用
   - 詳細説明は複数の観点を含める

3. **例**:
   ```
   コード品質: [utils.ts:42] エラーハンドリング不足 (High)
   偽陰性: [utils.ts:42] 例外を握りつぶす可能性 (Critical)

   統合結果: [utils.ts:42] 例外を握りつぶす可能性（エラーハンドリング不足） (Critical)
   ```

### Step 5: 優先順位の決定

すべての指摘事項を優先順位順にソート：

1. **ソート順**
   - 第1キー: 重要度 (Critical > High > Medium > Low)
   - 第2キー: カテゴリ (セキュリティ > ロジック > 品質 > スタイル)
   - 第3キー: ファイルパス（アルファベット順）

2. **アクション優先順位**
   - **P0（最優先）**: Critical な問題
   - **P1（高）**: High な問題
   - **P2（中）**: Medium な問題
   - **P3（低）**: Low な問題

### Step 6: エグゼクティブサマリーの作成

レビュー全体の要約を3-5文で作成：

1. **総合評価**（1文）
   - 例: 「全体として、コード品質は良好ですが、いくつかの重要な問題が検出されました。」

2. **主要な問題点**（1-2文）
   - 例: 「特に、エラーハンドリングとテストカバレッジに改善の余地があります。」

3. **推奨アクション**（1-2文）
   - 例: 「Critical な問題2件を最優先で修正し、テストカバレッジを80%以上に引き上げることを推奨します。」

### Step 7: 最終レポートの生成

`final-report.md` を以下の構成で生成：

1. **ヘッダー**: タイトル、PR情報
2. **総合判定**: APPROVED / NEEDS_FIX
3. **エグゼクティブサマリー**: 全体の要約
4. **レビューサマリー**: 4つのレビューそれぞれの概要
5. **統合指摘事項**: 優先順位順に整理された指摘リスト
6. **推奨アクション**: 次のステップ

### Step 8: 出力と確認

1. **ファイル出力**
   - `.tmp/review-results/final-report.md` に出力

2. **出力確認**
   - ファイルが正しいパスに存在するか
   - ファイルの内容が正しくフォーマットされているか
   - すべてのセクションが含まれているか

## 矛盾する指摘の調整

複数のレビューで矛盾する指摘がある場合の対応：

### パターン1: コード品質 vs 過剰実装
- **コード品質**: 「抽象化が不足している」
- **過剰実装**: 「抽象化が過剰である」

**調整方法**:
- 両方の観点を記載
- コンテキストに応じた判断を推奨
- 例: 「現時点では抽象化が過剰だが、将来の拡張性を考慮する場合は検討の余地あり」

### パターン2: テスト品質 vs 過剰実装
- **テスト品質**: 「テストケースが不足している」
- **過剰実装**: 「不要なテストケースがある」

**調整方法**:
- 具体的なケースを確認
- 必要なテストと不要なテストを区別
- 優先順位をつけて記載

### パターン3: 偽陰性 vs コード品質
- **偽陰性**: 「null チェックが不足」
- **コード品質**: 「エラーハンドリングが冗長」

**調整方法**:
- より厳しい方を採用（この場合は偽陰性の指摘）
- セキュリティや安全性を優先

## 最終確認チェックリスト

- [ ] 4つのレビュー結果がすべて読み込まれているか
- [ ] 総合判定が正しく決定されているか
- [ ] 重複する指摘が適切に統合されているか
- [ ] 指摘事項が優先順位順にソートされているか
- [ ] エグゼクティブサマリーが簡潔で明確か
- [ ] 推奨アクションが具体的で実行可能か
- [ ] 最終レポートが正しいパスに出力されているか
