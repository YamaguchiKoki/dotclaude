# コード品質レビュー詳細手順

## 事前確認

1. **レビュー対象範囲の特定**
   - `git status` で変更ファイル一覧を取得
   - `git diff` で変更内容を確認（unstaged changes）
   - `git diff --staged` でステージングされた変更を確認
   - 必要に応じて `git diff main...HEAD` でブランチ全体の変更を確認
   - レビュー対象外のファイル（generated、vendor、lockファイル等）を除外

2. **関連Skillsの読み込み**
   - `CLAUDE.md` から該当プロジェクトのコーディング規約を確認
   - 命名規則、コーディングスタイル、アーキテクチャ関連のSkillsを読み込む

## レビュー実施チェックリスト

### 1. 命名規則チェック

- [ ] 変数名が意図を明確に表現しているか
- [ ] 関数名が動作を正確に表現しているか
- [ ] クラス名が責務を適切に表現しているか
- [ ] 定数名がすべて大文字（UPPER_SNAKE_CASE）か
- [ ] プライベートメンバーの命名規則（_prefix等）が統一されているか
- [ ] 略語の使用が適切か（過度な短縮がないか）
- [ ] ドメイン用語が一貫して使用されているか

### 2. コードスタイルチェック

- [ ] インデントが統一されているか（スペース/タブ、幅）
- [ ] 行の長さが規定内か（通常80-120文字）
- [ ] 空行の使い方が適切か（論理的な区切り）
- [ ] コメントのスタイルが統一されているか
- [ ] インポート文の順序が適切か（標準→サードパーティ→ローカル）
- [ ] 不要なコメントアウトコードがないか
- [ ] TODO/FIXME コメントが適切に管理されているか

### 3. アーキテクチャ適合性チェック

- [ ] **Clean Architecture層間依存**
  - ドメイン層が外部依存を持っていないか
  - アプリケーション層がインフラ層に直接依存していないか
  - 依存性逆転の原則が適用されているか（インターフェース経由の依存）
- [ ] **DIコンテナ設定**
  - 新しいリポジトリやサービスがDIコンテナに登録されているか
  - スコープ（Singleton、Scoped、Transient）が適切か
  - 循環依存が発生していないか
- [ ] **ドメインロジックの配置**
  - ビジネスロジックがドメイン層に集約されているか
  - ドメインモデルが貧血モデルになっていないか
  - アプリケーション層が単なるオーケストレーションに留まっているか

### 4. コードの可読性

- [ ] 関数/メソッドが単一責任を持っているか（SRP）
- [ ] ネストの深さが適切か（3階層以内が理想）
- [ ] 早期リターン（Early Return）が活用されているか
- [ ] マジックナンバーが定数化されているか
- [ ] 複雑な条件式が変数やメソッドに抽出されているか
- [ ] コードの重複（DRY原則違反）がないか

### 5. エラーハンドリング

- [ ] 適切な例外型が使用されているか
- [ ] エラーメッセージが明確で有用か
- [ ] リソースの解放が適切に行われているか（try-finally、using等）
- [ ] エラーログが適切に出力されているか

## レビュー結果の出力

### 指摘事項の分類

各指摘事項を以下の重要度で分類：

- **Critical**: 致命的な問題（動作に影響、セキュリティリスク）
- **High**: 重要な問題（保守性の大幅な低下、パフォーマンス問題）
- **Medium**: 改善推奨（コード品質の向上）
- **Low**: マイナーな指摘（スタイルの統一等）

### 出力形式

`.tmp/review-results/code-quality.md` に以下の形式で出力：

```markdown
# コード品質レビュー結果

## 判定
- [APPROVED / NEEDS_FIX]

## サマリー
- 指摘事項数: X件
- 重要度内訳: Critical: X, High: X, Medium: X, Low: X

## 指摘事項

### [ファイルパス:行番号] タイトル
**重要度**: [Critical/High/Medium/Low]
**カテゴリ**: [命名規則/コードスタイル/アーキテクチャ/可読性/エラーハンドリング]

詳細説明

**推奨対応**:
修正方法の提案
```

## 最終確認

- [ ] すべての指摘事項が明確で実行可能か
- [ ] 重要度の判定が適切か
- [ ] 推奨対応が具体的か
- [ ] ファイルが正しいパスに出力されているか
