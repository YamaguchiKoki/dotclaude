# テスト品質レビュー詳細手順

## 事前確認

1. **テストファイルの特定**
   - `git status` でテストファイルの変更を確認（`*_test.*`, `*.test.*`, `*.spec.*`, `test_*.py` 等）
   - `git diff` で新規追加・変更されたテストファイルを確認
   - 対応する実装コードを確認

2. **テストフレームワークの確認**
   - 使用しているテストフレームワーク（Jest、pytest、RSpec等）を特定
   - プロジェクトのテスト規約を `CLAUDE.md` から確認

## レビュー実施チェックリスト

### 1. テストカバレッジチェック

- [ ] **正常系のカバー**
  - 主要な処理フローがテストされているか
  - 期待される入力に対する期待される出力が検証されているか
- [ ] **異常系のカバー**
  - エラーケースがテストされているか
  - 例外がスローされるケースが検証されているか
  - バリデーションエラーのテストがあるか
- [ ] **境界値のカバー**
  - 最小値・最大値のテストがあるか
  - 空データ（空配列、空文字列、null/undefined）のテストがあるか
  - オフバイワンエラーを検出できるテストがあるか
- [ ] **新規実装のテスト**
  - 追加された関数・メソッドにテストが存在するか
  - 変更された関数・メソッドのテストが更新されているか

### 2. テストケースの妥当性チェック

- [ ] **テストの目的が明確か**
  - テスト名（describe/it/test）が何をテストしているか明確か
  - Given-When-Then（Arrange-Act-Assert）の構造が明確か
- [ ] **適切なアサーション**
  - アサーションが具体的で意味のあるものか
  - 複数の概念を1つのテストで検証していないか
  - アサーションが多すぎないか（1テスト1アサーションが理想）
- [ ] **テストデータの品質**
  - テストデータが現実的か
  - マジックナンバーやハードコードされた値が適切に定数化されているか
  - ファクトリやビルダーパターンが適切に使用されているか

### 3. テストの独立性チェック

- [ ] **実行順序の独立性**
  - テストの実行順序に依存していないか
  - 各テストが独立して実行可能か
- [ ] **グローバル状態の管理**
  - グローバル変数やシングルトンの状態が適切にリセットされているか
  - setup/teardown（beforeEach/afterEach）が適切に実装されているか
- [ ] **テスト間の依存**
  - あるテストの結果が他のテストに影響しないか
  - 共有フィクスチャが適切に管理されているか

### 4. モック・スタブの適切性チェック

- [ ] **モックの使用目的**
  - 外部依存（DB、API、ファイルシステム等）が適切にモック化されているか
  - モックが過度に使用されていないか（実際のオブジェクトを使うべき箇所）
- [ ] **モックの精度**
  - モックの振る舞いが実際の実装を正確に模倣しているか
  - モックの戻り値が現実的か
- [ ] **スパイの活用**
  - 関数呼び出しの検証が適切に行われているか
  - 引数の検証が具体的か

### 5. テスト実行効率チェック

- [ ] **実行速度**
  - 不必要に遅いテストがないか（sleep、長いタイムアウト等）
  - 外部依存が適切にモック化されているか
- [ ] **テストの粒度**
  - 単体テストが細かすぎないか（内部実装の詳細をテストしていないか）
  - 統合テストが粗すぎないか（複数のコンポーネントをテストしすぎていないか）
- [ ] **重複の排除**
  - 同じロジックを複数のテストで重複して検証していないか
  - 共通の前処理がヘルパー関数に抽出されているか

### 6. テストコードの保守性チェック

- [ ] **可読性**
  - テストコードが読みやすいか
  - 複雑な条件やループが避けられているか
  - コメントが必要最小限か
- [ ] **保守性**
  - テストコードが実装コードと同じ品質基準を満たしているか
  - 重複コードが適切にリファクタリングされているか
  - テストユーティリティやヘルパーが適切に整理されているか

## テストカバレッジの計測

可能であれば、以下のコマンドでカバレッジを計測：

- **JavaScript/TypeScript**: `npm test -- --coverage` または `jest --coverage`
- **Python**: `pytest --cov`
- **Go**: `go test -cover`
- **Ruby**: `rspec --format documentation`

目標カバレッジ：
- 単体テスト: 80%以上
- 重要なビジネスロジック: 90%以上

## レビュー結果の出力

### 指摘事項の分類

各指摘事項を以下の重要度で分類：

- **Critical**: カバレッジの重大な欠如、誤ったテストロジック
- **High**: 重要な異常系のテスト不足、テストの独立性の問題
- **Medium**: 境界値テストの不足、モックの不適切な使用
- **Low**: テスト名の改善、軽微なコードスタイル

### 出力形式

`.tmp/review-results/test-quality.md` に以下の形式で出力：

```markdown
# テスト品質レビュー結果

## 判定
- [APPROVED / NEEDS_FIX]

## テストカバレッジ
- 全体カバレッジ: X%
- 新規追加コードのカバレッジ: X%

## サマリー
- 指摘事項数: X件
- 重要度内訳: Critical: X, High: X, Medium: X, Low: X

## 指摘事項

### [ファイルパス:行番号] タイトル
**重要度**: [Critical/High/Medium/Low]
**カテゴリ**: [カバレッジ/妥当性/独立性/モック/実行効率/保守性]

詳細説明

**推奨対応**:
修正方法の提案（具体的なテストケースの追加等）
```

## 最終確認

- [ ] 重要なロジックのテストカバレッジが十分か
- [ ] テストが実装の詳細ではなく振る舞いを検証しているか
- [ ] すべての指摘事項が具体的で実行可能か
- [ ] ファイルが正しいパスに出力されているか
