# 過剰実装レビュー

過剰実装（Over-Engineering）の検出レビューを実行してください。

## レビュー対象範囲

`git status` と `git diff` でレビュー対象ファイルを特定すること。
**変更されていない、レビュー対象外のファイルに対しレビューを行うことは固く禁じる。**

対象となる変更:
- `git diff`: unstaged な変更
- `git diff --staged`: staged な変更
- `git diff main...HEAD`: ブランチ全体の変更（必要に応じて）

## 実行手順

1. `git status` と `git diff` でレビュー対象ファイルを特定する
2. 対象ファイルの実装内容を確認し、要件定義やタスクの目的を理解する
3. 以下のレビュー観点に基づき、各対象ファイルをレビューする
4. 指摘事項を `.tmp/review-results/over-implementation.md` に出力する
5. `.tmp/review-results/over-implementation.md` に出力されているかチェックする。**もし正しいパスに出力できていなければ出力をやり直す**

## レビュー観点

### 1. YAGNI原則（You Aren't Gonna Need It）違反

- **未使用の抽象化**
  - 現時点で使われていない抽象クラスやインターフェースがないか
  - 将来の拡張性を見越した、現在は不要な設計がないか
  - 1箇所でしか使われていないのに抽象化されているコードがないか
- **不要な柔軟性**
  - 設定ファイルやパラメータが過度に多くないか
  - 現在の要件で不要なカスタマイズポイントがないか
  - プラグイン機構など、現時点で必要ない拡張機能がないか
- **過度な汎用化**
  - 特定用途のコードが不必要に汎用化されていないか
  - ユーティリティ関数が必要以上に多機能になっていないか
  - ジェネリクスや型パラメータが過度に使われていないか

### 2. 複雑すぎる設計パターン

- **デザインパターンの乱用**
  - 問題の規模に対してデザインパターンが過剰でないか
  - Factory、Builder、Strategy などのパターンが本当に必要か
  - シンプルな実装で十分なのに複雑なパターンを使っていないか
- **過度なレイヤー分割**
  - レイヤーが必要以上に多層化していないか
  - 単純な処理に対して中間層が多すぎないか
  - データ変換だけのためのレイヤーがないか

### 3. 不要な依存関係

- **ライブラリ・パッケージの追加**
  - 新しい依存関係が本当に必要か
  - 既存のライブラリや標準機能で代替可能ではないか
  - 小さな機能のために大きなライブラリを追加していないか
- **内部依存の複雑化**
  - モジュール間の依存関係が複雑すぎないか
  - 循環依存が発生していないか

### 4. 必要最小限の実装かどうか

- **スコープの確認**
  - タスクやIssueで求められている機能以外の実装がないか
  - 「ついでに」追加された機能がないか
  - 現在のPRの目的に集中しているか
- **コードの量**
  - 同じ結果をより少ないコードで実現できないか
  - 重複したロジックがないか
  - 冗長な条件分岐や処理がないか
- **設定の複雑さ**
  - 設定項目が必要最小限か
  - デフォルト値で十分なのに設定可能にしていないか

### 5. 早すぎる最適化

- **パフォーマンス最適化**
  - 計測なしにパフォーマンス最適化を行っていないか
  - 現時点でパフォーマンス問題が存在しないのに最適化していないか
  - 可読性を犠牲にした最適化がないか
- **キャッシング・メモ化**
  - 本当に必要なキャッシングか
  - キャッシュ管理の複雑さに見合う効果があるか

### 6. ドキュメント・コメントの過剰性

- **過度なドキュメント**
  - 自明なコードに対する冗長なコメントがないか
  - コードを読めば分かることをコメントで説明していないか
- **不要なTODO/FIXME**
  - 将来の拡張のためのTODOコメントが多すぎないか

## 判断基準

以下の質問に「No」なら過剰実装の可能性：

1. **今必要か？** - 将来のためだけの実装ではないか
2. **シンプルにできるか？** - より簡潔な方法はないか
3. **削除できるか？** - なくても動作するコードではないか
4. **読みやすいか？** - 複雑さに見合う価値があるか

## 各個のコメント形式

`pr-review-comment` skill の「インラインコメント」の項に従うこと。

## パイプライン出力

- **出力形式**: instruction で指定された出力形式に従うこと
- **出力先**: `.tmp/review-results/over-implementation.md`
- **注意**: ローカルレビューのため、リモートへのコメント投稿は行わない
