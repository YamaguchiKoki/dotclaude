# テスト品質レビュー

テストの品質レビューを実行してください。

## レビュー対象範囲

`git status` と `git diff` でレビュー対象ファイルを特定すること。
**変更されていない、レビュー対象外のファイルに対しレビューを行うことは固く禁じる。**

対象となる変更:
- `git diff`: unstaged な変更
- `git diff --staged`: staged な変更
- `git diff main...HEAD`: ブランチ全体の変更（必要に応じて）

## 実行手順

1. `git status` と `git diff` でレビュー対象ファイルを特定する
2. テストファイル（`*_test.*`, `*.test.*`, `*.spec.*`, `test_*.py` など）を中心にレビューする
3. 以下のレビュー観点に基づき、各対象ファイルをレビューする
4. 指摘事項を `.tmp/review-results/test-quality.md` に出力する
5. `.tmp/review-results/test-quality.md` に出力されているかチェックする。**もし正しいパスに出力できていなければ出力をやり直す**

## レビュー観点

### 1. テストカバレッジ

- **単体テスト（Unit Test）のカバレッジ**
  - 主要な関数・メソッドに対するテストが存在するか
  - 正常系（Happy Path）がテストされているか
  - 異常系（Error Path）がテストされているか
- **統合テスト（Integration Test）のカバレッジ**
  - コンポーネント間の連携が適切にテストされているか
  - 外部依存（DB、API等）との統合テストが適切か

### 2. テストケースの妥当性

- **境界値テスト**
  - 最小値、最大値、境界付近の値がテストされているか
  - 空配列、空文字列、null/undefined のケースがカバーされているか
- **異常系テスト**
  - エラーハンドリングが適切にテストされているか
  - 例外スローのケースが網羅されているか
  - バリデーションエラーのテストがあるか
- **エッジケース**
  - レアケースや特殊なシナリオがテストされているか
  - 並行処理、タイミング依存のケースが考慮されているか

### 3. テストコードの保守性

- **テストの可読性**
  - テストケース名が明確で、何をテストしているか理解しやすいか
  - Arrange-Act-Assert（AAA）パターンが適切に使われているか
  - テストコードが簡潔で、冗長な記述がないか
- **テストの独立性**
  - 各テストが独立して実行可能か
  - テスト間の依存関係がないか
  - グローバル状態の変更が適切にクリーンアップされているか
- **テストデータの管理**
  - テストデータ（fixtures）が適切に管理されているか
  - マジックナンバーやハードコードされた値が適切に定数化されているか

### 4. モック・スタブの適切性

- **モックの使用**
  - 外部依存がモック化されているか
  - モックが過度に使用されていないか（実際のオブジェクトを使うべき箇所）
  - モックの設定が正確で、実際の動作を適切に模倣しているか
- **スタブ・フェイクの使用**
  - データベースやAPIのスタブが適切に実装されているか
  - テスト用のフェイクオブジェクトが有効活用されているか

### 5. テスト実行効率

- **テスト実行速度**
  - 不必要に遅いテストがないか
  - テストの並列実行が考慮されているか
- **テストの重複**
  - 同じロジックを重複してテストしていないか
  - テストの粒度が適切か（過度に細かい、または粗いテストがないか）

## 各個のコメント形式

`pr-review-comment` skill の「インラインコメント」の項に従うこと。

## パイプライン出力

- **出力形式**: instruction で指定された出力形式に従うこと
- **出力先**: `.tmp/review-results/test-quality.md`
- **注意**: ローカルレビューのため、リモートへのコメント投稿は行わない
