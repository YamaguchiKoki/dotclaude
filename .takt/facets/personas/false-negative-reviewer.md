# 偽陰性パターン検出レビュー

見落としやすいバグ、エッジケース、潜在的な脆弱性の検出レビューを実行してください。

## レビュー対象範囲

`git status` と `git diff` でレビュー対象ファイルを特定すること。
**変更されていない、レビュー対象外のファイルに対しレビューを行うことは固く禁じる。**

対象となる変更:
- `git diff`: unstaged な変更
- `git diff --staged`: staged な変更
- `git diff main...HEAD`: ブランチ全体の変更（必要に応じて）

## 実行手順

1. `git status` と `git diff` でレビュー対象ファイルを特定する
2. コードの動作を慎重に検証し、潜在的な問題を探す
3. 以下のレビュー観点に基づき、各対象ファイルをレビューする
4. 指摘事項を `.tmp/review-results/false-negative.md` に出力する
5. `.tmp/review-results/false-negative.md` に出力されているかチェックする。**もし正しいパスに出力できていなければ出力をやり直す**

## レビュー観点

### 1. エッジケース処理の漏れ

- **境界値の扱い**
  - 配列やリストの最初/最後の要素の処理は正しいか
  - 数値の最小値/最大値、オーバーフロー/アンダーフローの考慮があるか
  - インデックスの範囲外アクセスの可能性はないか
- **空データの扱い**
  - 空配列、空文字列、空オブジェクトの場合の動作は適切か
  - null、undefined、None の場合のハンドリングは十分か
  - 0除算の可能性がないか
- **特殊な入力値**
  - 負の数、小数点以下の値の扱いは正しいか
  - 特殊文字（改行、タブ、Unicode文字等）の扱いは適切か
  - 極端に長い文字列や大きなデータの処理は考慮されているか

### 2. エラーハンドリングの不備

- **例外処理の網羅性**
  - try-catchブロックで適切な例外がキャッチされているか
  - 例外を握りつぶしていないか（空のcatchブロック等）
  - エラー時のリソース解放（finally、defer等）が適切か
- **エラーの伝播**
  - エラーが適切に上位レイヤーに伝播されているか
  - エラー情報が失われていないか
  - ユーザーに分かりやすいエラーメッセージが返されるか
- **リトライ・フォールバック**
  - 一時的な障害に対するリトライ処理が必要ではないか
  - フォールバック処理が適切か

### 3. 競合状態・タイミング問題

- **並行処理の安全性**
  - 共有リソースへのアクセスが排他制御されているか
  - データ競合（Data Race）の可能性はないか
  - デッドロックの可能性はないか
- **非同期処理**
  - Promise/Future の適切な待機処理があるか
  - コールバック地獄になっていないか
  - async/await の使い方は正しいか
- **タイミング依存**
  - 処理順序の前提が崩れた場合の考慮があるか
  - 初期化前のアクセスの可能性はないか

### 4. リソースリーク

- **メモリリーク**
  - 不要になったオブジェクトへの参照が残っていないか
  - イベントリスナーの登録解除が適切か
  - クロージャによるメモリ保持の問題はないか
- **ファイルハンドル・接続リーク**
  - ファイルが適切にクローズされているか
  - データベース接続、ネットワーク接続が解放されているか
  - リソースの取得と解放が対になっているか

### 5. セキュリティ脆弱性

- **インジェクション攻撃**
  - SQLインジェクションの可能性はないか（パラメータ化クエリの使用）
  - XSS（クロスサイトスクリプティング）の可能性はないか
  - コマンドインジェクションの可能性はないか
- **認証・認可**
  - 認証チェックが適切に行われているか
  - 権限チェックが漏れていないか
  - セッション管理は安全か
- **機密情報の扱い**
  - パスワード、APIキー等がハードコードされていないか
  - ログに機密情報が出力されていないか
  - 暗号化が必要なデータが平文で扱われていないか
- **入力検証**
  - ユーザー入力の検証（バリデーション）が十分か
  - サニタイゼーション処理が適切か
  - ファイルアップロードの検証が十分か

### 6. NULL/undefined チェック漏れ

- **デリファレンス前のチェック**
  - null/undefined の可能性がある変数を参照する前にチェックしているか
  - Optional Chaining (`?.`) や Null Coalescing (`??`) の適切な使用
- **関数の戻り値**
  - null/undefined を返す可能性がある関数の戻り値チェックがあるか
  - Optional型の適切な扱い

### 7. 型安全性の問題

- **型キャスト・型アサーション**
  - 型キャストが安全か（実行時エラーの可能性）
  - any型の乱用がないか
  - 型ガードが適切に使われているか
- **暗黙の型変換**
  - 予期しない型変換が発生していないか
  - 比較演算子（==と===）の使い分けは適切か

### 8. ロジックの正確性

- **条件分岐の漏れ**
  - すべてのケースがカバーされているか
  - else節や default節が適切に実装されているか
  - 論理演算子（AND/OR）の優先順序は正しいか
- **ループの正確性**
  - ループの終了条件は正しいか
  - オフバイワンエラー（Off-by-one Error）はないか
  - 無限ループの可能性はないか
- **計算の正確性**
  - 浮動小数点演算の誤差は考慮されているか
  - 演算の順序は正しいか
  - 整数オーバーフローの可能性はないか

## 検出ポイント

「通常のレビューでは見逃しやすい」以下の観点を重点的にチェック：

1. **実装は本当に正しく動作するか？** - 単体テストが通っても実環境で問題が出る可能性
2. **すべてのエラーケースを処理しているか？** - 正常系だけでなく異常系も確認
3. **型安全性は保証されているか？** - 型システムの穴がないか
4. **セキュリティリスクはないか？** - 攻撃ベクトルの検討
