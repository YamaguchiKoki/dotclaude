# テストスコープ 出力フォーマット（ts-issue-test-scope）

以下の形式でテストスコープを出力してください。

---

# テストスコープ

## 作成したテストファイル一覧

| テストファイル | テスト種別 | 対象モジュール |
|--------------|----------|--------------|
| `packages/backend/src/__tests__/foo.test.ts` | EBT | `foo.ts` |
| `packages/frontend/src/__tests__/Bar.test.tsx` | EBT + PBT | `Bar.tsx` |

## 受け入れ条件 → テストの対応表

| 受け入れ条件 # | 受け入れ条件の内容 | テストファイル | テスト名 | テスト種別 |
|-------------|----------------|--------------|---------|----------|
| 1 | [受け入れ条件の説明] | `path/to/test.ts` | `describe > it 名` | EBT |
| 1 | [受け入れ条件の説明（PBT）] | `path/to/test.ts` | `property: ...` | PBT |
| 2 | [受け入れ条件の説明] | `path/to/test.ts` | `describe > it 名` | EBT |

## fast-check プロパティ一覧

| プロパティ名 | 対応する受け入れ条件 # | 検証する不変条件 |
|------------|-------------------|--------------|
| [プロパティ名] | 1 | [不変条件の説明] |

## RED 確認結果

```bash
# テスト実行コマンド
npm test

# 出力スニペット（失敗テストの一部）
# [テスト名] ✗ [失敗理由]
# ...
```

- 全テストが失敗（RED）していることを確認: [✅ 確認済み / ❌ 一部 GREEN のテストが存在（要調査）]

## ビルド確認結果

```
npx tsc --noEmit
# 出力: [エラーなし / エラー内容]
```

---

## 出力ルール

1. テストファイルは実際のパスを記述すること
2. 受け入れ条件 # は `issue-plan.md` の番号と一致させること
3. PBT は fast-check の `fc.property(...)` を使って記述すること
4. 1つの受け入れ条件に複数のテスト（EBT + PBT）がある場合は複数行に記述する
5. RED 確認結果（テスト実行コマンド・出力スニペット・全テスト失敗確認の ✅）を必ず記録する
6. ビルド（型チェック）の結果を必ず記録する
