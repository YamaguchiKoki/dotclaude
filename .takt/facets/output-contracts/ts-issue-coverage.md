# 実装カバレッジ 出力フォーマット（ts-issue-coverage）

以下の形式で実装カバレッジを出力してください。

---

# 実装カバレッジ

## 受け入れ条件 → テスト → 実装の対応表

| 受け入れ条件 # | 受け入れ条件 | テスト | 実装箇所 | 担保状況 |
|-------------|-----------|-------|---------|---------|
| 1 | [条件の説明] | `test.ts:L20` | `src/foo.ts:L10` | ✅ 担保済み |
| 2 | [条件の説明] | `test.ts:L40` | `src/bar.ts:L30` | ✅ 担保済み |
| 3 | [条件の説明] | なし | `src/baz.ts:L50` | ⚠️ 部分担保 |

## 未担保・部分担保の項目

担保できていない箇所がある場合は、その理由を記述してください：

| 受け入れ条件 # | 担保状況 | 理由 | 対応方針 |
|-------------|---------|------|---------|
| 3 | 部分担保 | [担保できていない理由] | [今後の対応方針] |

担保できていない項目がない場合:

> 全受け入れ条件が担保されています。

## 人間による動作確認手順（verification）

実装の動作を確認するための手順を記述してください：

### 環境起動

```bash
# 例: 開発サーバーの起動
npm run dev
```

### 確認手順

1. [操作手順 1]
2. [操作手順 2]
3. [期待される結果]

### E2E 確認（frontend のみ）

frontend を変更した場合は、ブラウザでの動作確認手順を記述：

1. ブラウザで `http://localhost:3000` を開く
2. [具体的な操作手順]
3. [期待される表示・動作]

---

## 出力ルール

1. 受け入れ条件 # は `issue-plan.md` の番号と一致させること
2. テスト・実装箇所はファイルパスと行番号まで記述すること
3. 担保状況は「✅ 担保済み」「⚠️ 部分担保」「❌ 未担保」の3種類で表現する
4. 未担保がある場合は必ず理由を記述する（「未対応」だけでは不可）
5. 動作確認手順は実際に実行可能な具体的な手順を記述すること
